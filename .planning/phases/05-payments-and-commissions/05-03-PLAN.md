Extend the existing payments service with these new functions:

1. `adminApprovePayment(orderId: string)`:
   - Find order by id. Validate status is 'PENDING'.
   - Find or create Payment record for this order (may not exist if PIX was never initiated).
   - Call `processPaymentConfirmation(orderId)` — same function used by webhook.
   - Update Payment record: `paidAt: new Date()`, `providerStatus: 'approved'`, `method: 'PIX'`.
   - Return the updated order.

2. `adminRejectPayment(orderId: string)`:
   - Find order. Validate status is 'PENDING'.
   - Transition order to 'CANCELLED' and all PENDING items to 'CANCELLED'. Use `$transaction`.
   - Delete or mark Payment record with providerStatus 'rejected'.
   - Return the updated order.

3. `refundPayment(paymentId: string, amount?: number)`:
   - Find payment. Validate it has a providerPaymentId and paidAt (it was actually paid).
   - If Mercado Pago configured: call `mpPayment.refund({ id: providerPaymentId })` for full refund or construct partial refund with `amount` parameter. Wrap in try/catch — if MP call fails, log error but still update local record (admin can retry).
   - Update Payment: increment `refundedAmount` by refund amount, set `providerStatus` to 'refunded' (full) or keep as 'approved' (partial).
   - If full refund: transition all order items to 'REFUNDED' status, transition order to 'CANCELLED'.
   - Return updated payment.

4. `getUserPaymentHistory(userId: string, page: number, limit: number)`:
   - Query Payment records joined with Order where `order.buyerId === userId`.
   - Include order details: orderNumber, status, totalAmount, createdAt.
   - Sort by createdAt desc. Paginate.
   - Return `{ payments, total, page, limit }`.

5. `adminListPendingPayments(page: number, limit: number)`:
   - Query orders with status 'PENDING' that have Payment records (PIX initiated but not confirmed).
   - Include payment data and buyer info (name, email).
   - Sort by order createdAt asc (oldest first — most urgent).
   - Paginate. Return `{ orders, total, page, limit }`.

6. `adminListAllPayments(filters: { status?: string, page: number, limit: number })`:
   - Query all Payment records with order and buyer info.
   - Optional filter by providerStatus.
   - Sort by createdAt desc. Paginate.

Add routes to `payments.routes.ts`:
- `GET /history` — authenticated, call `getUserPaymentHistory`, return paginated.
- `POST /admin/approve` — authenticate + authorize('ADMIN'), validate body with `adminApprovePaymentSchema`, call `adminApprovePayment`.
- `POST /admin/reject` — authenticate + authorize('ADMIN'), validate body `{ orderId: z.string() }`, call `adminRejectPayment`.
- `POST /:paymentId/refund` — authenticate + authorize('ADMIN'), validate body with `refundPaymentSchema`, call `refundPayment`.
- `GET /admin/pending` — authenticate + authorize('ADMIN'), validate query with paginationSchema, call `adminListPendingPayments`.
- `GET /admin/list` — authenticate + authorize('ADMIN'), validate query with `listPaymentsSchema`, call `adminListAllPayments`.

IMPORTANT: Place static routes (`/history`, `/admin/*`) BEFORE parameterized routes (`/:orderId/*`) in Express router.

Extend the existing commission service with dashboard aggregation:

1. Add to `commission.service.ts`:

   a. `getCommissionDashboard(periodStart: Date, periodEnd: Date)`:
      - Total commissions in period: SUM of `orderItem.commissionAmountSnapshot` where order status is in ('PAID', 'PROCESSING', 'SHIPPED', 'DELIVERED', 'COMPLETED') and `order.createdAt` is in range.
      - Total sales volume: SUM of `orderItem.priceSnapshot`.
      - Transaction count.
      - Use Prisma `$queryRaw` for the aggregation (more efficient than loading all records):
        ```sql
        SELECT
          oi.commission_rate_snapshot as rate,
          COUNT(*) as transaction_count,
          SUM(oi.commission_amount_snapshot) as total_commission,
          SUM(oi.price_snapshot) as total_sales
        FROM order_items oi
        INNER JOIN orders o ON o.id = oi.order_id
        WHERE o.status IN ('PAID','PROCESSING','SHIPPED','DELIVERED','COMPLETED')
          AND o.created_at >= ?
          AND o.created_at <= ?
        GROUP BY oi.commission_rate_snapshot
        ```
      - Return structured result: `{ byPlan: [...], totals: { totalCommission, totalSales, transactionCount }, period: { start, end } }`.

   b. `getCommissionTransactions(page: number, limit: number, periodStart?: Date, periodEnd?: Date)`:
      - List individual order items with commission data, joined with order (orderNumber, buyerId, status) and collection item -> catalog entry (title).
      - Filter by date range if provided. Only include paid/completed orders.
      - Sort by order createdAt desc. Paginate.
      - Return `{ transactions, total, page, limit }`.

2. Add routes to `commission.routes.ts`:
   - `GET /admin/dashboard` — authenticate + authorize('ADMIN'), query params: `periodStart` (ISO date string, required), `periodEnd` (ISO date string, required). Call `getCommissionDashboard`, return data.
   - `GET /admin/transactions` — authenticate + authorize('ADMIN'), query params: paginationSchema + optional `periodStart`, `periodEnd`. Call `getCommissionTransactions`, return paginated.

   Create a `commissionDashboardSchema` in the route file (or in contracts):
   ```
   z.object({
     periodStart: z.string().datetime({ offset: true }).or(z.string().date()),
     periodEnd: z.string().datetime({ offset: true }).or(z.string().date()),
   })
   ```

   And `commissionTransactionsSchema`:
   ```
   paginationSchema.extend({
     periodStart: z.string().optional(),
     periodEnd: z.string().optional(),
   })
   ```

Note: Place admin routes before any existing parameterized routes in the router.
