# Plan 08-01: Dispute API — Contracts + CRUD + Evidence Upload

## Wave: 1 (no dependencies)

## Goal
Create the dispute creation endpoint with time window validation, evidence upload, dispute listing for buyer/seller, and Zod contract schemas.

## Tasks

### 1. Create contracts schemas (`packages/contracts/src/disputes.ts`)
- `createDisputeSchema` — orderItemId, reason (enum), description (min 10, max 5000)
- `submitDisputeResponseSchema` — message (min 10, max 5000)
- `resolveDisputeSchema` — status (RESOLVED_REFUND | RESOLVED_PARTIAL_REFUND | RESOLVED_NO_REFUND), resolution (min 10, max 5000), refundAmount (optional positive number)
- `listDisputesSchema` — extends paginationSchema with status filter, role filter
- `addDisputeMessageSchema` — message (min 1, max 5000)
- Export all types + add to `packages/contracts/src/index.ts`

### 2. Create dispute service (`apps/api/src/modules/disputes/disputes.service.ts`)
- `createDispute(buyerId, data)` — Inside $transaction:
  - Validate OrderItem exists, belongs to buyer
  - Validate OrderItem status is SHIPPED or DELIVERED
  - Check time window: 7 days post-delivery OR 30 days if not delivered (from shippedAt)
  - Check no existing OPEN/IN_MEDIATION dispute on same OrderItem
  - Create Dispute record with status OPEN
  - Update OrderItem.status to DISPUTED
  - Update Order.status to DISPUTED (if not already)
  - Fire notification to seller (DISPUTE_OPENED)
  - Return dispute with relations

- `getDispute(disputeId, userId, role)` — Fetch dispute with evidence + messages, verify access (buyer, seller, or admin)

- `listBuyerDisputes(buyerId, filters, pagination)` — Buyer's disputes with pagination

- `listSellerDisputes(sellerId, filters, pagination)` — Seller's disputes with pagination

- `addEvidence(disputeId, userId, file, description)` — Upload image via Cloudinary, create DisputeEvidence record. Max 5 evidence per party per dispute.

- `cancelDispute(disputeId, buyerId)` — Buyer cancels, revert OrderItem status to previous

### 3. Create dispute routes (`apps/api/src/modules/disputes/disputes.routes.ts`)
- `POST /` — createDispute (authenticated, buyer only by logic)
- `GET /my/buyer` — listBuyerDisputes (authenticated)
- `GET /my/seller` — listSellerDisputes (authenticated)
- `GET /:id` — getDispute (authenticated, access-controlled)
- `POST /:id/evidence` — addEvidence (authenticated, multer single image)
- `POST /:id/cancel` — cancelDispute (authenticated, buyer only)

### 4. Register routes in create-app.ts
- Import and mount at `/api/v1/disputes`

### 5. Rebuild contracts
- `pnpm --filter contracts build`

## Verification
- POST creates dispute with valid time window
- POST rejects dispute outside time window
- POST rejects duplicate dispute on same OrderItem
- GET returns dispute with evidence and messages
- Evidence upload works (Cloudinary or local fallback)
- Cancel reverts OrderItem status
